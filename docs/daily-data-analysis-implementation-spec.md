# 日足分析 — 実装仕様書（レビュー用ドラフト）

`daily-data-analysis-proposals.md` の Top10 を中心に、**実装方法**・**トレード判断に役立つ描画方法**・**拡張メモ**を分析ごとに詳述する。実装着手前の確認用ドラフト。最後の「§A 確認したい設計判断」を先に決めてから着手したい。

凡例: 各節は `目的の問い / lib（計算層）/ 描画仕様 / UI操作 / Guide要点 / 統合先 / 拡張・確認` で構成。
描画は既存 `CandleSeasonalityChart` の Canvas2D 規約（`initCanvas` + DPR、`layout/frame/drawXLabels` 系のヘルパ）に揃える。

## §0 共通設計方針（全分析に効く土台）

### 0.1 「今の状態」を必ず画面に出す — *Nowマーカー*
本カタログの分析は大半が「状態 → 先行きの分布」を出す。トレード判断に直結させる鍵は、
**過去の集計だけでなく『今日（最新足）がどのバケット/分位にいるか』を常にハイライトする**こと。

- 棒グラフ/ヒートマップ: 最新足が属するバケットを**太枠＋▼マーカー＋「現在」ラベル**で強調。
- 時系列: 最新値に水平線と数値ピル。
- これを各コンポーネント共通の描画ヘルパ `drawNowMarker(ctx, x, y, label)` として切り出す。

> 設計理念: 「過去はこうだった」で終わらせず「**だから今日はこう構える**」まで一画面で言い切る。

### 0.2 統計的有意性の共通表示 — *StatBadge*
アノマリー系（1.1/5.1/9.1/8.1 等）は数値の隣に必ず:
- `n`（標本数）, `平均±95%CI`（ブートストラップ or t）, `p値`, FDR後の有意フラグ。
- 共通の小コンポーネント `<StatBadge n p ciLow ciHigh significant />` を作り、有意なら緑、非有意/n<30 は灰色で「参考」と明示。
- 既存 `WeekdayEdgeScanChart` の検定ロジック（ブート/FDR）を `app/lib/stats-significance.ts` として抽出・共用する。

### 0.3 条件付き先行きリターンの共通エンジン — `conditional-forward-returns.ts`
9.1 / 5.1 / 1.1 / 4.1 / 4.4 / 6.x / 10.x は**同一骨格**（状態でバケット化 → N日先リターンの平均・勝率・分散・有意性・年次持続性）。これを共通ライブラリ化し、各分析は「状態関数 `stateOf(i): label`」を渡すだけにする。

```ts
// app/lib/conditional-forward-returns.ts
export interface ForwardStats {
  label: string; n: number;
  meanFwd: number; medianFwd: number; winRate: number; stdFwd: number;
  ciLow: number; ciHigh: number; p: number; significant: boolean; // FDR後
  byYear: { year: number; meanFwd: number; n: number }[]; // 持続性
}
export function conditionalForwardReturns(
  prices: PricePoint[],
  stateOf: (i: number) => string | null,  // null=除外
  horizon: number,                          // N日先
  opts?: { entry?: "close" | "open"; logReturn?: boolean; boot?: number }
): ForwardStats[];
```

- フォワードリターン定義: `r = (C[i+N] - entry[i]) / entry[i]`（`entry` は当日 close か翌日 open）。先読みバイアス回避のため、状態は「i 日終値時点で確定する情報のみ」を使う。
- 多重比較: バケット数ぶんの p値を Benjamini-Hochberg FDR で補正。
- 持続性: 年ごとに `meanFwd` を再集計し、符号が安定して同じかを別途可視化。

**これを最初に作ると 6本以上の分析が薄いラッパで済む**（§A で着手順を確認したい）。

## §1 状態 → 先行きリターン表（提案 9.1）★最優先

- **問い**: 「いまのこの状態（RSI帯・ボラレジーム・MA距離・トレンド）で、N日先は平均何%・勝率何%か。それは有意で、今も生きているか」
- **lib**: `conditional-forward-returns.ts`（§0.3）。状態関数を複数定義:
  - `rsiBucket`(RSI(14) を <30/30-50/50-70/>70 等), `volRegime`(YZボラ分位で低/中/高), `maDist`((C-SMA200)/SMA200 を分位), `trendState`(Kaufman効率比 §4.3 ＋傾き符号).
- **描画**:
  1. **メイン表**（HTML table、Canvasでなく可読性優先）: 行=状態バケット、列=`n / 平均N日 / 勝率 / 分散 / 95%CI / 有意`。
     - セル背景を平均リターンで赤緑ヒートマップ（diverging, 0中心）。
     - 勝率セルは 50% 基準でバー内包（ミニ横バー）。
     - **今日が属する行を太枠＋「◀ 現在」**（§0.1 Nowマーカー）。
  2. **横バー比較**（Canvas）: 各バケットの平均N日リターンを 0中心 diverging 棒（既存 `drawDiverging` 流用）。有意バケットのみ濃色、非有意は半透明。
  3. **持続性ストリップ**: 各バケット×年の符号ヒートマップ（既存 WeekdayEdgeScan の年次ヒートマップ流用）。
- **UI**: 状態軸セレクタ（RSI/ボラ/MA距離/トレンド）、ホライズン N スライダ（1/5/10/20日）、エントリー（引け/翌寄り）トグル。
- **Guide要点**: 条件付き期待値の意味、フォワードリターン定義、勝率と期待値の違い、先読みバイアス、FDR、年次持続の見方。
- **統合先**: 新セクション候補 `conditional`（§A.1 で確認）。当面は `regime` か `calendar` 末尾に追加でも可。
- **拡張・確認**: 状態軸を 2変数化したのが 9.2（下記）。まず1変数表で土台を固め、9.2 をヒートマップ拡張として乗せる。

### §1b 2変数コンディショニング・ヒートマップ（提案 9.2）

## §2 OHLCボラ推定（Yang-Zhang ほか）（提案 2.1）

- **問い**: 「終値だけで測るより、どれだけ正確に1日の真の変動を測れているか。今の本当の日次リスクはどれか」
- **lib**: `app/lib/ohlc-volatility.ts`
  ```ts
  export interface VolEstimates { close:number; parkinson:number; gk:number; rs:number; yangZhang:number; }
  export function rollingOHLCVol(prices: PricePoint[], window: number): { time:string; est:VolEstimates }[];
  export function efficiencyRatio(prices: PricePoint[], window:number): Record<keyof VolEstimates, number>; // close法分散比
  ```
  - Parkinson: `(1/4ln2)·mean(ln(H/L)²)`
  - Garman-Klass: `mean(0.5 ln(H/L)² − (2ln2−1) ln(C/O)²)`（既存 candle-seasonality に式あり、再利用）
  - Rogers-Satchell: `mean(ln(H/C)ln(H/O) + ln(L/C)ln(L/O))`（ドリフトに頑健）
  - **Yang-Zhang**: `σ²_overnight + k·σ²_open-close + σ²_RS`、`k=0.34/(1.34+(n+1)/(n-1))`（窓込みで最良効率）
  - 全て年率化（√252）してσ%で返す。
- **描画**:
  1. **ローリングσ時系列**（lightweight-charts LineSeries 複数）: close/YZ/GK を重ね描き。YZ を主役色、close を細い灰で「終値法だとこう過小評価」が見えるように。最新値ピル＋Nowライン。
  2. **効率比バー**（Canvas）: 各推定量の「close法に対する分散削減率」を横バー（YZが一番長い＝効率的、を一目で）。
  3. **現在ボラの数値カード**: 「YZ日次σ ◯%／年率◯%／close法比 −◯%」。
- **UI**: 窓長スライダ（10/20/60日）、年率化トグル。
- **Guide要点**: 各推定量の前提（Parkinson=ドリフト0・窓無視、GK=窓無視、RS=ドリフト可、YZ=窓込み）、効率比の意味、いつどれを使うか。
- **統合先**: `ohlc`（OHLC分析）か `volatility`。GARCH/VolCone の入力として後段に波及。
- **拡張・確認**: 2.3 HAR と 2.2 GARCH入力差し替えはこのσを観測値に使う。HAR（`RV_t=β0+βd·RVd+βw·RVw+βm·RVm`）は別コンポーネントで、YZ実現ボラを入力に回帰＋予測線。

## §3 オーバーナイト vs 日中エクイティ（提案 8.1）

- **問い**: 「この銘柄のリターンは“夜間の持ち越し”で稼いでいるのか、“日中”で稼いでいるのか。どちらに賭けるべきか」

  ```ts
  // overnight_t = (O_t - C_{t-1}) / C_{t-1}   intraday_t = (C_t - O_t)/O_t
  export interface DecompEquity { time:string; overnight:number; intraday:number; buyhold:number; } // 累積(複利)
  export function decomposeEquity(prices: PricePoint[]): DecompEquity[];
  export function decompStats(prices: PricePoint[]): {

  ```
- **描画**:
  1. **3本の累積エクイティ曲線**（lightweight-charts, 対数軸オプション）: 夜間（引け買い→寄り売り）／日中（寄り買い→引け売り）／単純保有。色で明快に分け、最終リターンをピル表示。**どちらの線が右肩上がりか＝戦略の答え**。
  2. **寄与サマリーバー**: 総リターン・シャープ・分散を夜間/日中で積み上げ or 並置。「リスクの◯%は夜間が占める」を強調（持ち越しリスクの定量化＝提案 8.3 も内包）。

- **UI**: 対数/線形軸トグル、期間追従（既存 period 連動）。
- **Guide要点**: overnight/intraday の定義、オーバーナイト・ドリフト異常（多くの株指数で夜間にリターン集中する実証）、なぜ起きるか（リスクプレミアム/流動性）、持ち越しリスクとの両睨み。
- **統合先**: `transform` か `risk`。`SpiralHeatmap` と相性が良いので `calendar` でも可。
- **拡張・確認**: 取引コスト未控除だと夜間戦略は過大評価されがち（毎日2回約定）。Guide に明記し、将来 §4 のスプレッド推定でコスト控除版を重ねたい。

## §4 高安スプレッド推定（Corwin-Schultz / Abdi-Ranaldo）（提案 3.2）

- **問い**: 「板データ無しで、この銘柄の実効取引コスト/流動性はどれくらいか。地合いで流動性が枯れていないか」
- **lib**: `app/lib/spread-estimator.ts`
  - **Corwin-Schultz (2012)**: 連続2日の H/L から。`β=E[(ln(H_t/L_t))²+(ln(H_{t+1}/L_{t+1}))²]`, `γ=(ln(H_{t,t+1}/L_{t,t+1}))²`（2日通算の高安）, `α=(√(2β)−√β)/(3−2√2) − √(γ/(3−2√2))`, `S=2(e^α−1)/(1+e^α)`。負値は0クリップ。
  - **Abdi-Ranaldo (2017)**: midrange `η=(ln H+ln L)/2` を用いた頑健版。
  - ローリング（月次など）でスプレッド% 時系列を返す。
- **描画**:
  1. **実効スプレッド%の時系列**（LineSeries）: 上昇＝流動性悪化。価格急落局面でスパイクするのが見え、**スプレッド急拡大＝地合い悪化シグナル**。Nowライン＋直近平均比。
  2. **コスト換算カード**: 「現在の推定実効スプレッド ◯bp ＝ 往復コスト ◯bp」。バックテストの現実的コスト控除に使える旨。
  3. （任意）Amihud非流動性（提案 3.1）を第2軸で重ねると流動性ビューが完成。
- **UI**: 推定法トグル（CS/AR）、窓長、bp/％表示。
- **Guide要点**: 高安からスプレッドを逆算する原理（分散はレンジに、スプレッドは2日共通成分に現れる）、各手法の頑健性差、負値クリップ、出来高が薄い銘柄での誤差。
- **統合先**: `risk` か新規 `liquidity`（§A.1）。
- **拡張・確認**: Amihud（3.1）/RVOL（3.3）と束ねて「流動性パネル」化するか、スプレッド単体で出すか（§A で確認）。

## §9 保有期間別 MFE/MAE → 最適 TP/SL（提案 13.1）

- **問い**: 「エントリー後、利確・損切りをどこに置けば期待値が最大になるか」
- **lib**: 既存 `MFEMAEChart` 拡張。各エントリー後の経路から保有日数別の MFE（最大含み益）・MAE（最大含み損）分布を作り、**TP/SL グリッド上の期待値**を計算。
  ```ts
  // entry条件ごとに、TP×SL の格子で期待R・勝率・期待値を評価
  export function tpSlSurface(prices, entrySignal, opts): { tp:number; sl:number; expR:number; winRate:number }[];
  ```
- **描画**:
  1. **TP×SL 期待値ヒートマップ**（Canvas格子）: X=利確幅(%/ATR)、Y=損切幅。色=期待R。**最大セルに★マーカー**＝推奨TP/SL。
  2. **MFE/MAE分布ヒスト**: 保有N日時点の含み益/含み損の分布。「利確はMFE中央値の手前に」が見える。
