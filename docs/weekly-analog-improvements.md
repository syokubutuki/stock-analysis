# 今週の軌跡アナログ — 改善提案

本書は実装済みの「今週の軌跡アナログ比較」（個別: `cal-weekly-analog` / 横断: `pf-weekly-analog-cross` / コア: `app/lib/weekly-analog.ts`）に対する改善提案を、**統計的妥当性 → 手法の質 → UX/性能** の順に優先度付きで整理したものである。

各項目に **区分**（【新規】/【拡張: 対象】）・**問題**・**計算根拠**・**活用**・**優先度**(★1〜3) を付す。

> 規約: 新規コンポーネントは必ず `AnalysisGuide` で手法・数式・用語・読み方・活用・注意点を解説すること（`CLAUDE.md`）。

---

## 0. 現状の実装と、その前提の棚卸し

### 0.1 実装済み

| 層 | 内容 |
|---|---|
| コア | `computeWeeklyAnalog()` — 全窓を「窓末=今日=0%」に再基準化。リードイン(t<0)＋フォワード(t>0)を連続表現 |
| 絞り方 | `similar`(z化形状のkNN) / `usbin`(窓起点=週初めの前夜米国ビン) |
| 距離 | `euclid`(等速) / `dtw`(Sakoe-Chibaバンド=窓長の約1/4) |
| 窓 | `trailing`(直近L営業日) / `week`(月曜起点・L=今週の経過日数、過去は各週の先頭L日) |
| HL | `fwdHigh/fwdLow` = running max/min（MFE/MAE）、中央値パスを緑赤点線で表示 |
| 出力 | 終値中央値・平均・勝率・MFE/MAE中央値・25–75%帯・事例一覧 |
| 横断 | 全銘柄一斉適用＋「N銘柄中M銘柄が上向き」コンセンサス。MFE/MAE/勝率/事例でソート |
| 日中 | 今週側のみ60分足ドリルダウン（履歴制約により過去アナログ週は再現不可） |

### 0.2 現状が暗黙に置いている前提（＝改善の起点）

1. **「事例N件」を独立N標本として扱っている** → 実際は窓が重複しており独立でない
2. **絶対値で判断させている**（中央値+0.8%・勝率58%）→ 無条件ベースラインとの比較がない
3. **アナログ予測に予測力がある**ことを検証していない → in-sample の記述統計にすぎない
4. **上位K件を等重み**で平均 → 遠い近傍も同じ1票
5. **z化で水準・ボラを吸収** → 静かな週と荒い週が同じ形として一致しうる（値幅の意味が変わる）
6. **横断コンセンサスで銘柄を独立票として数えている** → 実際は同一地合いで強く相関

> **総括**: 「見せ方」は完成しているが、**「そのシグナルが本物か」を判定する手段が UI に存在しない**。以下 A 群はこの穴を塞ぐもので、他のどの改善よりも優先すべきである。

### A1. 無条件ベースラインとの差＋有意性検定 ★★★

**問題**: 現在の表示は「その後H日の中央値 +0.8% / 勝率 58%」。だがその銘柄の**無条件の**H日リターンが中央値 +0.7% / 勝率 56% なら、このアナログは何も語っていない。ユーザーは絶対値を見て「勝率58%＝有望」と誤読する。

**計算根拠**:
- ベースライン: 選抜に使わなかった**全候補窓**（または全立会日起点）のH日フォワード分布。中央値 `m₀`、勝率 `p₀`。
- 差の検定は**並べ替え検定(permutation)**が素直（分布仮定不要・既存 `session-gap.ts` に `permTestDiff` の前例あり）:
  1. 実測差 `Δ = m_sel − m₀`
  2. 候補窓のフォワードをシャッフルし、同サイズ `n_sel` を無作為抽出して差 `Δ*` を計算、B=2000回
  3. `p = (#{|Δ*| ≥ |Δ|} + 1) / (B + 1)`
  4. **重要**: シャッフルは A2 のブロック単位で行う（重複窓の相関を保存するため）
- 勝率は二項検定 or 同じ並べ替えで `p₀` との差を検定。

**UI**: バナーを「中央値 +0.8%（ベースライン +0.7% / **差 +0.1pt, p=0.62**）」形式に。★表記は既存 `StatBadge` / `star()` の慣習に合わせる。ベースライン中央値パスを**灰色の細線**でチャートに重ねると一目で分かる。

**活用**: 「地合いで絞った意味があったか」が初めて判定できる。差が有意でないビンは見送る。

---

### A2. 重複窓の実効標本数＋ブロックブートストラップ ★★★

**問題**: `trailing` モードでは候補窓末 `j` と `j+1` はリードインを L−1 日共有し、**フォワードも H−1 日共有する**。よって隣接事例はほぼ同じ観測。「事例300」の独立数は実質 `≈ 300/H` 程度。25–75%帯もCIも**過度に狭く**、有意性を過大評価する。

`week` アラインはリードイン重複を解消するが、**H>5 ではフォワードが依然重複**する（隣接週の先行き5日が翌週と重なる）。

**計算根拠**:
- **実効標本数**: フォワードH日が重なる事例は1ブロックとみなす。単純近似 `n_eff ≈ n_sel / H`（重複度から算出）か、既存 `intraday-basket.ts` の `clusterStat`（日付クラスタ頑健SE）の考え方を日付ブロックに転用。
- **ブロックブートストラップ**: i.i.d. リサンプリングは相関を壊す。長さ `b ≈ H` の**移動ブロック**で窓末インデックスをリサンプリングし、各リサンプルで中央値を再計算 → 中央値のCIを得る。
  - ブロック長は `b = H`（フォワード重複長）が自然。
- 表示: 事例数の隣に **`実効n`** を併記（`WeekdayUsCross` の `nEff` 併記と同じ慣習）。

**活用**: 「事例300」に安心していたユーザーが「実効60」を見てサイズを抑える。A1の検定もブロック単位で行い、初めて正しいp値になる。

### A3. ウォークフォワードOOS検証 ★★★

**問題**: **この機能で最も重要な欠落**。「似た形のあと上がった」は過去の記述であって、**予測力の証拠ではない**。現状ユーザーはそれを予測として使う導線に乗っている。

**計算根拠**（新規 `weekly-analog-oos.ts` ＋ `WeeklyAnalogOosChart`）:
1. 検証期間の各週末 `t`（例: 直近3年の各週）について:
   - **`t` 以前のデータだけ**でアナログを構築（候補窓末 `< t − H` に制限し、フォワードのリークも防ぐ）
   - 予測 `ŷ_t` = 選抜事例のフォワード中央値
   - 実測 `y_t` = 実際のH日先リターン
2. 指標:
   - **IC(情報係数)**: `ρ_Spearman(ŷ, y)`。週次で `n≈150`(3年)。
   - **方向的中率**: `P(sign(ŷ) = sign(y))` を無条件 base rate と比較
   - **分位バケット**: `ŷ` を5分位に分け、各バケットの実測平均 → **単調性**があるか
   - IC のブートストラップCI（A2同様ブロック単位）
3. 設定ごと（mode/metric/align/L/H）に IC を並べ、**どの設定が本当に効くか**を提示。

**活用**: 「アナログは効くのか？」に初めて答える。ICが0近辺なら、この機能は**先読みではなく文脈提示**として使うべきと分かる（それ自体が正しい結論）。**多重比較の罠**に注意 — 設定を総当たりしてIC最良を選ぶのは過学習。既存 `app/lib/walk-forward.ts`（`runWalkForward` が **DSR**=Deflated Sharpe Ratio(試行数補正) と **PBO**=Probability of Backtest Overfitting を算出）の考え方を流用し、試行数＝設定カタログ数として補正する。

### A4. 横断コンセンサスの相関調整 ★★

**問題**: 「10銘柄中8銘柄が上向き → 横断的に強気」。だが同一市場の銘柄は相関0.5〜0.8。8/10 は8つの独立証拠ではなく、**実質1〜2票**に近い。現状の文言は過信を招く。

**計算根拠**:
- 銘柄の予測 `ŷ_i` の平均に対し、**横断相関を吸収したSE**を使う。既存 `intraday-basket.ts` の `clusterStat`（日付クラスタ頑健）と同じ発想で、日付クラスタ = 同一週。
- 簡易には実効銘柄数 `N_eff = N / (1 + (N−1)·ρ̄)`（`ρ̄` = 銘柄間フォワード相関の平均）。ρ̄=0.6, N=10 → `N_eff ≈ 1.6`。
- 表示: 「8/10銘柄が上向き（**実効 1.6銘柄相当** — ほぼ地合い1票）」。

**活用**: 「全銘柄が同方向＝強い証拠」ではなく「**全銘柄が同方向＝分散が効いていない**」という正しい読みに誘導できる（既に注記はあるが、数値で示すべき）。

### B1. カーネル重み付け＋棄却 ★★★

**問題**: `similar` は上位K件を**等重み**。1位（距離0.3）と20位（距離2.5）が同じ1票。さらに**「似た局面が存在しない」場合でも必ずK件返す** — 今週が前例のない形でも、無理やり集めた20件で予測を出してしまう。

**計算根拠**:
- **Nadaraya-Watson**: `ŷ = Σ w_i·y_i / Σ w_i`, `w_i = exp(−d_i² / (2h²))`（ガウスカーネル）。バンド幅 `h` は距離の中央値等で自動設定。
- 重み付き中央値/分位も同様（重み付き経験分布から算出）。
- **棄却**: 最近傍距離 `d_min` が過去の距離分布の上位X%なら「**今週は前例が薄い（novelty高）**」と警告し、予測を出さない or 信頼度を下げる。
- 副産物として **距離パーセンタイル**表示（「今週の形は過去に対して上位12%の珍しさ」）は単体でも有用。

**活用**: 「似ていない事例で薄めた予測」を避ける。棄却状態自体が「今週は前例なし＝定石が効かない」という情報。

---

### B2. ボラ正規化 ★★

**問題**: `zShape()` は形状比較のため水準とスケールを両方消す。結果、**σ=0.5%の静かな週**と**σ=3%の荒い週**が「同じ形」として一致しうる。だがフォワードの値幅は当然まったく違う。MFE/MAE中央値を「利確目安」として出している以上、この不整合は実害がある。

**計算根拠**（いずれか/併用）:
1. **σ単位で出力**: フォワードを `y / σ_t`（`σ_t`=窓期間の実現ボラ）で標準化して集計し、表示時に**今週のσ**を掛け戻す。「似た形のあと **+1.2σ** 上げた → 今のσなら +1.5%」。
2. **ボラ整合フィルタ**: 候補を `σ_cand / σ_query ∈ [0.6, 1.6]` に限定。
3. 距離にボラ比の項を加える: `d' = d_shape + λ·|ln(σ_cand/σ_query)|`。
- σは既存の Yang-Zhang / Parkinson（`docs/daily-data-analysis-proposals.md` 2.1）を流用すればOHLCを活かせる。

**活用**: MFE/MAEの値幅目安が現在のボラ環境に整合する。低ボラ期に高ボラ期の値幅を期待する誤りを防ぐ。

---

### B3. DTWバンド可変＋経路長正規化 ★★

**問題**: 現状バンドは `max(1, round(L*0.25))` 固定。バンドはDTWの結果を大きく左右する（広いほど「何でも似ている」に漸近）のに、ユーザーが触れず、感度も見えない。また累積コストを**warping経路長で正規化していない**ため、経路長の異なる候補間で厳密には非可換。

**計算根拠**:
- バンド `w` をUI公開（0=等速≡ユークリッド 〜 L）。`w` を動かすと近傍集合がどう変わるかを見せる。
- 経路長正規化: DP で経路長 `Λ(i,j)` も同時に持ち、`d = √(D(n,m) / Λ(n,m))`。全候補が同じ窓長なら影響は小さいが、厳密性のため。
- **警告表示**: `w ≥ L/2` では退化（1点が多数点に対応）が起きやすい旨を注記。

---

### B4. usbin ∩ similar アンサンブル ★★

**計算根拠**: 「前夜米国が大幅高で始まり、**かつ**形も似ている」週に絞る（積集合）。または重み積 `w = w_bin · w_shape`。事例は減るのでA2の実効n表示が必須。

**活用**: 地合いと形が両立した局面は確度が高い。両モードで結論が食い違う場合の裁定にもなる。

---

### B5. 銘柄横断プール・アナログ ★★

**問題**: 自己履歴10年＝約500週。週境界アライン×5分位ビンだと各≈100週、`similar`のK=20と合わせると薄い。

**計算根拠**: ウォッチリスト/業種バスケットの**他銘柄の過去週もアナログ候補に含める**（既存 `intraday-basket.ts` / `SectorBasketWeekdayChart` のプーリング思想の日足版）。リターンは銘柄ごとにボラ正規化（B2）してから距離計算・集計。**同一週の複数銘柄は1クラスタ**として A2 のブロックに含める（横断相関を吸収）。

**活用**: 事例数を数倍に増やし、薄いセルを救う。ただし「銘柄が似た反応をする」前提が必要（`WeekdayUsCross` の注意点と同じ）。

---

### B6. 距離にHLを含める ★

**計算根拠**: 現状の距離は終値パスのみ。`leadHigh/leadLow` は既に保持しているので、レンジ列 `(H−L)/C` の z化系列を第2チャネルとし、`d = √(d_close² + γ·d_range²)`。「同じ終値経路でも、荒れながら来たのか静かに来たのか」を区別できる。

### C1. 信頼度バッジ ★★★

**問題**: 現状ユーザーは「事例数・帯幅・距離」を自分で総合判断せねばならない。A1〜A3を実装しても、指標が増えるほど**読み手の統合負荷**が上がる。

**計算根拠**: 4条件のAND的スコア → 3段階表示。
- 実効n（A2）が閾値以上（例 `n_eff ≥ 20`）
- ベースライン差が有意（A1, `p < 0.05`）
- 最近傍距離が棄却域でない（B1）
- OOSのIC（A3）がその設定で正（設定単位のキャッシュ参照）

→ **緑=採用可 / 黄=参考 / 赤=使うな**。既存 `StatBadge` の意匠を流用。

**活用**: 「この画面の数字を信じてよいか」が一目で分かる。**A群の価値を実際の判断に接続する最後の一手**。

### C3. Web Worker化の見積り

`computeWeeklyAnalog` は候補数 `≈2500`(10y) × `L` × DTWバンド `w`。L=20/w=5 なら約 `2500×20×11 ≈ 55万` 演算/銘柄。横断20銘柄で **≈1100万演算/設定変更** → メインスレッドで数百ms〜秒級。既存 `sarima.worker.ts` の前例に倣い Worker 化するのが素直（`app/lib/weekly-analog.worker.ts`）。まず計測、次に最適化の順で。

## D. 実装順の推奨

```
Phase 1（正直さの回復・これ単独で価値が出る）
  A1 ベースライン＋並べ替え検定
  A2 実効n＋ブロックブートストラップ   ← A1のp値の前提。同時に入れる
  C1 信頼度バッジ（A1/A2の範囲で先行版）

Phase 2（予測力の測定）
  A3 ウォークフォワードOOS（＋walk-forward.ts の DSR/PBO で多重比較調整）
  A4 横断コンセンサスの相関調整

Phase 3（手法の質）
  B1 カーネル重み＋棄却/novelty（C5同梱）
  B2 ボラ正規化
  B3 DTWバンド可変

Phase 4（拡張）
  B4 アンサンブル / B5 横断プール / B6 HL距離
  C2 ジャンプ / C3 Worker / C4 永続化 / C6 ホバー
```

**A1+A2 を最優先とする理由**: 現状のUIは「勝率58%」を根拠に発注しうる導線を持ちながら、それがベースライン+2ptの誤差範囲かどうかを**確かめる手段がない**。機能追加より先に、既に出している数字の意味を正すべきである。A3 で「実は予測力がない」と出たとしても、それは失敗ではなく**この機能を"文脈提示ツール"として正しく位置づけ直す**という価値ある結論になる。
